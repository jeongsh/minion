# 경기 일정 관리 유기성 개선 작업 지시서

## 1. 목적

현재 경기 관리 기능은 매치, 세트, 밴픽, 선수 스탯, 타임라인이 순차적으로 추가되면서 각 기능은 동작하지만 서로의 상태를 일관되게 보정하지 못한다.

이번 작업의 목표는 다음과 같다.

- 세트 결과가 저장되면 매치 스코어, 상태, 최종 승자가 자동으로 맞춰진다.
- 매치 정보가 바뀌면 하위 세트가 허용되지 않는 팀이나 번호를 갖지 못한다.
- 경기 데이터 동기화 작업을 운영자가 한 번에 이해하고 실행할 수 있다.
- 자동 동기화 값과 수동 보정 값의 우선순위를 명확히 한다.
- 관리자 화면의 버튼 수를 줄이고 작업 단계를 명확하게 만든다.
- 기존 불일치 데이터를 안전하게 탐지하고 복구할 수 있다.

이 문서는 구현자가 별도 맥락 없이 바로 작업할 수 있도록 현재 문제, 목표 모델, UI 구조, 백엔드 작업, 데이터 정리, 검증 기준을 한 파일에 정리한다.

## 2. 현재 확인된 상태

### 2.1 실제 사례

관리 화면:

- `/admin/matches/2026 Mid-Season Invitational_Play-In Day 4_1/edit`
- 매치: T1 vs TL, Bo5
- 매치 상태: `scheduled`
- 매치 스코어: `null : null`
- 매치 승자: 없음
- 1세트 상태: `data_synced`
- 1세트 승자: T1
- 1세트 결과, 밴픽, 선수 10명 상세 스탯 존재

즉, 하위 세트에는 경기 결과가 있지만 상위 매치는 여전히 경기 전으로 저장되어 있다.

### 2.2 2026-07-01 Supabase 읽기 전용 점검 결과

- 매치: 420건
- 세트: 965건
- 현재 `deriveSetStatus()` 규칙으로 다시 계산했을 때 상태 불일치 세트: 448건
  - `finished -> data_synced`: 390건
  - `draft_done -> finished`: 39건
  - `draft_done -> data_synced`: 4건
  - `scheduled -> finished`: 3건
  - `scheduled -> data_synced`: 12건
- 승자가 있지만 `scheduled`인 세트: 15건
- 결과 또는 상세 데이터가 있지만 결과 이전 상태인 세트: 58건
- KC vs TL 한 경기에는 다음 모순이 동시에 존재
  - 저장 스코어: KC 3 : 0 TL
  - 저장 승자: TL
  - 세트 승자 집계: KC 0 : 3 TL

### 2.3 구조적 원인

- `matches.team_a_score`, `matches.team_b_score`, `matches.status`, `matches.winner_team_id`가 세트와 별도로 저장된다.
- 세트 수동 생성/수정 후 상위 매치를 다시 계산하지 않는다.
- 전체 경기 일정 동기화와 개별 매치 세트 동기화가 별도 흐름이다.
- 증분 경기 동기화는 마지막 종료 경기 시각 이후 일정만 조회하므로 이미 저장된 과거 `scheduled` 경기가 결과 갱신 대상에서 빠질 수 있다.
- 화면별 경기 상태 판정 기준이 다르다.
  - 일정/경기 상세: 저장된 `match.status`
  - 홈 일부 영역: 경기 시작 시각으로 `live` 추론
  - 팬 페이지: 저장 상태와 경기 시각을 혼용
- DB에는 상태, 승자, 참가 팀 간의 관계를 보장하는 충분한 제약이 없다.

예외가 필요하면 일반 입력 필드가 아니라 `수동 보정` 흐름으로 분리하고, 보정 이유와 시각을 기록한다.

## 5. 반드시 지켜야 하는 불변조건

### 5.1 매치

- 팀 A와 팀 B는 서로 달라야 한다.
- 매치 승자는 팀 A 또는 팀 B 중 하나여야 한다.
- 스코어는 0 이상의 정수여야 한다.
- 세트 결과가 하나도 없으면 기본 상태는 `scheduled`다.
- 한 세트 이상 끝났지만 과반승이 확정되지 않았으면 `live`다.
- Bo3는 2승, Bo5는 3승 도달 시 `completed`다.
- `completed`이면 최종 승자와 양 팀 스코어가 있어야 한다.
- 세트 데이터가 존재하면 매치 스코어와 승자는 세트 집계와 일치해야 한다.

기권, 몰수, 취소 경기를 지원해야 한다면 일반 경기 규칙을 느슨하게 만들지 말고 별도 `result_type` 또는 예외 상태를 추가한다.

### 5.2 세트

- 블루팀과 레드팀은 서로 달라야 한다.
- 블루팀과 레드팀은 상위 매치 참가 팀 두 팀이어야 한다.
- 세트 승자는 블루팀 또는 레드팀 중 하나여야 한다.
- 세트 번호는 1 이상이며 같은 매치 안에서 중복될 수 없다.
- 일반적인 경우 세트 번호는 `best_of`보다 클 수 없다.
- 결과 상태인 `finished`, `data_synced`에는 승자가 있어야 한다.
- `data_synced`는 선수 스탯 일부가 아니라 완전성 조건을 만족해야 한다.
  - 기본 조건: 양 팀 5명, 총 10명
  - 선수/팀/포지션 중복 및 누락 없음
- 밴픽 완료는 실제 기대 픽/밴 개수를 기준으로 판정한다.

## 6. 핵심 도메인 함수

### 6.1 `reconcileMatchFromSets(matchId)`

공통 함수 또는 DB RPC로 구현한다. 모든 세트 변경 경로는 마지막에 이 함수를 호출해야 한다.

입력:

- 매치 ID

처리:

1. 매치의 팀 A/B, `best_of` 조회
2. 해당 매치의 결과가 확정된 세트 조회
3. 팀 A/B 승수 집계
4. 매치 상태 계산
5. 스코어, 상태, 최종 승자를 한 번에 갱신
6. 계산 결과와 저장 결과 반환

상태 계산 예시:

```ts
if (finishedSetCount === 0) {
  status = matchHasStartedByTrustedSignal ? "live" : "scheduled";
  winnerTeamId = null;
} else if (Math.max(winsA, winsB) >= winsNeeded) {
  status = "completed";
  winnerTeamId = winsA > winsB ? teamAId : teamBId;
} else {
  status = "live";
  winnerTeamId = null;
}
```

중요:

- 재실행해도 같은 결과가 나오는 멱등 함수여야 한다.
- 결과 정정 시 `completed -> live`, 승자 제거, 스코어 감소도 가능해야 한다.
- 기존 값과 달라진 필드를 반환해 관리자 UI에서 요약할 수 있어야 한다.

### 6.2 `deriveSetStatus()` 보강

현재 규칙은 선수 스탯 한 건만 있어도 `data_synced`로 판정한다. 아래처럼 완전성 기준을 명시한다.

```ts
scheduled
  -> draft_in_progress: 밴픽 일부 존재
  -> draft_done: 기대 픽/밴 완성
  -> finished: 승자와 기본 결과 존재
  -> data_synced: finished 조건 + 유효한 선수 스탯 10명
```

상태는 가능한 한 데이터로 계산한다. 수동 상태 선택은 예외 보정 UI 안으로 이동한다.

## 7. 모든 변경 경로 연결

다음 경로에서 세트 변경이 성공하면 반드시 같은 트랜잭션 또는 안전한 후속 단계로 매치를 재조정한다.

- `createSetAction`
- `updateSetAction`
- `syncLeaguepediaMatchSets`
- 세트 삭제 기능이 추가될 경우 해당 삭제 액션
- 데이터 복구/백필 스크립트

세트가 다른 매치로 이동할 수 있다면 이전 매치와 새 매치를 모두 재조정한다. 가능하면 기존 세트의 `match_id` 변경은 별도 이동 기능이 아니면 막는다.

매치의 팀 A/B 또는 `best_of`를 변경할 때 하위 세트가 이미 있으면 다음 중 하나를 적용한다.

- 기본: 변경 차단 후 충돌 세트 안내
- 명시적인 마이그레이션 기능: 영향받는 세트를 보여주고 확인 후 변경

조용히 기존 세트의 의미를 바꾸면 안 된다.

## 8. 외부 데이터 동기화 구조

### 8.1 통합 액션

매치 관리 화면의 기본 동기화는 `경기 데이터 동기화` 하나로 제공한다.

권장 순서:

1. Leaguepedia MatchSchedule 최신 정보 조회
2. 세트 결과 upsert
3. 밴픽 upsert
4. 선수 스탯 upsert
5. 세트 상태 재계산
6. 매치 스코어/상태/승자 재계산
7. Game ID가 준비된 세트의 타임라인 동기화
8. 불일치 검사
9. 변경 요약 반환

### 8.2 원본 우선순위

- 세트 결과가 있으면 매치 스코어/승자는 세트 집계 우선
- 세트 결과가 없을 때만 MatchSchedule 스코어를 fallback으로 사용
- `venue`, `vod_url`, 공식 POM 같은 수동 관리 값은 외부 값이 비어 있다고 `null`로 덮어쓰지 않는다.
- 자동값과 수동값이 충돌하면 조용히 덮어쓰지 말고 경고 또는 보정 기록을 남긴다.

### 8.3 증분 동기화

마지막 종료 경기 시각보다 큰 일정만 조회하는 strict cursor 방식은 기존 경기 결과 갱신을 놓칠 수 있다.

다음 범위를 매번 재조회한다.

- 최근 7일 또는 대회 특성에 맞는 lookback 구간
- 모든 미래 일정
- 아직 `scheduled` 또는 `live`인 과거 경기

동기화는 upsert 기반으로 멱등하게 만든다.

### 8.4 트랜잭션과 데이터 보호

현재 일부 흐름은 기존 밴픽/선수 스탯을 삭제한 뒤 새 값을 삽입한다. 중간 실패 시 데이터가 사라질 수 있다.

- 가능한 경우 DB RPC 또는 트랜잭션 사용
- 새 payload를 먼저 검증
- 임시/staging 데이터 또는 원자적 교체 사용
- 외부 응답이 비어 있거나 불완전하면 기존 데이터를 삭제하지 않음
- 부분 성공을 단순 실패로 숨기지 말고 단계별 결과 반환

## 9. 관리자 매치 화면 개편

대상:

- `app/admin/matches/[matchId]/edit/page.tsx`
- `app/admin/matches/match-fields.tsx`
- 동기화 버튼 컴포넌트

### 9.1 상단

링크만 유지:

- `← 경기 목록`
- `공개 페이지`

버튼처럼 강하게 보이지 않는 보조 링크 스타일을 사용한다.

### 9.2 경기 기본정보

직접 편집:

- 경기명
- 일시
- 대회/스테이지
- 팀 A/B
- Best of
- 공식 POM
- VOD
- 경기장
- Leaguepedia Match ID

집계 표시 또는 읽기 전용:

- 현재 상태
- 팀 A/B 스코어
- 최종 승자

수동 보정이 필요하면 접힌 `고급 설정 > 결과 수동 보정` 안에서 제공한다.

기본 버튼:

- `기본정보 저장`

변경사항이 없으면 비활성화하고, 저장되지 않은 변경 여부를 보여준다.

### 9.3 경기 데이터 영역

주 버튼 하나:

- `경기 데이터 동기화`

보조 메뉴 `...`:

- 세트 결과만 다시 가져오기
- 모든 세트 타임라인 다시 가져오기
- 강제 전체 갱신
- 데이터 일관성 재검사

`타임라인 불러오기`는 세트 결과와 Game ID가 선행 조건이므로 기본 화면에서 세트 결과 동기화와 같은 레벨의 버튼으로 두지 않는다.

### 9.4 동기화 상태 표시

다음 칩 또는 요약을 보여준다.

- 일정
- 세트 결과
- 밴픽
- 선수 스탯
- 타임라인
- 마지막 동기화 시각

상태 예:

- 완료
- 일부 누락
- 준비 안 됨
- 실패
- 수동 보정 중

동기화 성공 메시지는 단순 건수뿐 아니라 다음을 보여준다.

```text
세트 1개 갱신
스코어 -:- -> 1:0
매치 상태 예정 -> 진행 중
밴픽 20개, 선수 스탯 10명, 타임라인 42개 저장
```

### 9.5 세트 목록

표시 컬럼:

- 세트 번호
- 상태
- 승자
- 진영
- 경기 시간
- 데이터 완성도
- 타임라인 유무

행 전체를 세트 편집 링크로 사용할 수 있다. 별도 `수정` 버튼은 제거하거나 아이콘 수준으로 낮춘다.

`세트 추가`는 다음 경우에만 주요하게 노출한다.

- Leaguepedia Match ID가 없는 수동 경기
- 외부에 아직 생성되지 않은 세트를 관리자가 직접 만들어야 하는 경우

일반 동기화 경기에서는 `... > 수동 세트 추가`로 낮춘다.

## 10. 관리자 세트 화면 개편

대상:

- `app/admin/matches/[matchId]/edit/sets/set-editor.tsx`
- `app/admin/sets/actions.ts`

### 10.1 상단 문맥

표시:

- `← 매치 관리`
- `T1 vs TL`
- `1세트`
- 현재 시리즈 스코어 `T1 1 : 0 TL`
- 매치 상태 `진행 중`

세트 상단 좌우의 점수는 현재처럼 매치 스코어를 의미 불명확한 `-`로 표시하지 않는다.

다음 두 정보를 분리한다.

- 세트 결과: `1세트 · T1 승리`
- 시리즈 결과: `T1 1 : 0 TL`

### 10.2 팀 선택

- 블루/레드 팀 선택지는 상위 매치 두 팀으로 제한
- 단순 진영 교체가 목적이면 두 개의 전체 팀 select 대신 `진영 교체` 액션 제공
- 세트 승자 선택지도 블루/레드 두 팀으로 제한

### 10.3 상태와 승자

기본은 데이터에서 계산한 읽기 전용 표시다.

- 예정
- 밴픽중
- 밴픽완료
- 경기종료
- 상세데이터 동기화

직접 변경은 `수동 보정`에서만 가능하게 한다. 수동 보정 시 이유 입력을 권장한다.

### 10.4 긴 폼 구성

한 페이지에 모든 입력을 계속 노출하지 않는다.

권장 탭 또는 접기 영역:

1. 결과 요약
2. 밴픽
3. 선수 스탯
4. 타임라인
5. 평점/리뷰 확인

`라인별 / 순서별`은 액션 버튼이 아니라 밴픽 영역 내부의 segmented control로 표시한다.

### 10.5 저장 버튼

현재 동일한 `Save set` submit 버튼이 두 개 있다. 하나의 고정 하단 액션 바로 통합한다.

구성:

- `변경사항 저장`
- `되돌리기`
- 저장되지 않은 변경사항 표시
- 저장 중/성공/실패 상태

버튼은 실제 변경이 있을 때만 활성화한다.

타임라인 재수집은 세트의 Game ID와 결과 상태가 준비됐을 때만 `... > 타임라인 다시 가져오기`로 제공할 수 있다.

## 11. DB 및 검증 제약

필요한 제약을 현재 스키마와 실제 예외 사례를 확인한 뒤 migration으로 추가한다.

후보:

- 매치 스코어 `>= 0`
- 팀 A와 팀 B가 서로 다름
- 블루팀과 레드팀이 서로 다름
- 결과 상태 세트에는 승자가 필요
- 매치/세트 승자가 참가 팀 안에 있는지 검증

다른 행을 조회해야 하는 참가 팀 검증은 단순 check constraint만으로 어렵다. 저장 서비스 또는 trigger/RPC에서 검증한다.

마이그레이션을 만들기 전에 기존 불일치 데이터를 먼저 정리하거나 `NOT VALID` 전략을 검토한다.

## 12. 기존 데이터 복구

별도 dry-run 가능한 스크립트를 만든다.

예시:

- `scripts/reconcile-match-set-status.ts`

옵션:

```text
--dry-run
--match=<uuid-or-leaguepedia-id>
--tournament=<id>
--all
```

dry-run 출력:

- 변경 대상 매치/세트 수
- 기존 값 -> 계산 값
- 스코어/승자 충돌
- 참가 팀 밖의 승자
- 세트 번호 이상
- 불완전 선수 스탯

적용 순서:

1. dry-run
2. 최근 MSI 사례 수동 검토
3. 좁은 매치 범위 적용
4. 재조회 검증
5. 전체 범위 적용
6. DB 제약 활성화

복구 시 `result_recorded_at`을 단순히 현재 시각으로 채우면 과거 세트의 평점 입력창이 다시 열릴 수 있다. 실제 경기 종료 시각 또는 안전한 별도 정책을 사용한다.

## 13. 캐시 및 화면 갱신

매치/세트 재조정 뒤 최소 다음 경로를 갱신한다.

- `/admin/matches`
- 해당 매치 편집 화면
- `/admin/sets`
- `/schedule`
- `/`
- `/matches/[matchId]`
- 해당 세트 상세
- 관련 팀 팬 경기 화면

가능하면 개별 `revalidatePath` 나열보다 경기 데이터용 cache tag를 도입한다.

## 14. 테스트 요구사항

현재 `package.json`에는 전용 테스트 스크립트가 없다. 이번 도메인 로직은 최소한 순수 함수 테스트가 필요하다.

### 14.1 매치 상태 전이

- Bo3, 세트 없음 -> `0:0`, `scheduled`, 승자 없음
- Bo3, A 1승 -> `1:0`, `live`, 승자 없음
- Bo3, A 2승 -> `2:0`, `completed`, A 승자
- Bo3, 1:1 -> `live`
- Bo5, 2:2 -> `live`
- Bo5, B 3승 -> `completed`, B 승자
- 완료 경기 결과 한 세트 제거 -> 상태/스코어/승자 하향 재계산
- 같은 함수를 두 번 실행해도 결과 동일

### 14.2 세트 상태

- 데이터 없음 -> `scheduled`
- 밴픽 일부 -> `draft_in_progress`
- 기대 밴픽 완성 -> `draft_done`
- 승자와 결과 존재 -> `finished`
- 유효 선수 스탯 10명 -> `data_synced`
- 선수 스탯 1~9명 -> `data_synced` 금지

### 14.3 검증

- 매치 참가 팀이 아닌 세트 팀 거부
- 블루/레드 동일 팀 거부
- 참가 팀이 아닌 승자 거부
- 음수 스코어 거부
- `best_of`보다 큰 세트 번호 처리
- 외부 빈 응답이 기존 밴픽/스탯을 삭제하지 않음

### 14.4 UI

- 기본 화면에서 주요 동기화 버튼은 하나
- 긴 세트 폼의 저장 액션은 하나
- 변경사항이 없으면 저장 버튼 비활성화
- 동기화 결과로 매치 스코어/상태 변경 내용 표시
- 불일치가 있으면 관리자에게 경고 표시
- 일반 사용자는 고급 강제 동기화/수동 보정을 실수로 실행하기 어려움

최종 검증:

```powershell
npm.cmd run typecheck
```

변경 파일 대상 scoped ESLint도 실행한다. 저장소 전체 lint의 기존 오류는 수정 범위와 분리해서 보고한다.

## 15. 구현 권장 순서

### 1단계: 도메인 로직

- `reconcileMatchFromSets()` 구현
- `deriveSetStatus()` 완전성 기준 보강
- 순수 함수 테스트

### 2단계: 쓰기 경로 연결

- 세트 생성/수정 후 매치 재조정
- Leaguepedia 세트 동기화 후 재조정
- 결과 정정/삭제 대응
- 필요한 트랜잭션 구성

### 3단계: 데이터 진단

- dry-run 스크립트
- 현재 MSI 사례 검증
- 448개 상태 불일치와 매치 충돌 목록 재확인

### 4단계: 매치 관리자 UI

- 집계 필드 읽기 전환
- 통합 동기화 버튼
- 보조 작업 메뉴
- 세트 데이터 완성도 표시

### 5단계: 세트 관리자 UI

- 상위 매치 문맥 표시
- 팀/승자 선택 범위 제한
- 저장 버튼 단일화
- 탭/접기 구조
- 수동 보정 분리

### 6단계: 데이터 복구 및 제약

- 좁은 범위 dry-run/적용
- 전체 데이터 복구
- DB 제약 추가
- 공개 화면 상태 일치 확인

## 16. 완료 조건

다음이 모두 충족되어야 완료다.

- T1 vs TL 사례에서 1세트 동기화 후 매치가 `1:0`, `live`로 자동 변경된다.
- Bo5에서 한 팀이 3승하면 매치가 자동으로 `completed`가 되고 승자가 확정된다.
- 세트 결과 정정 시 매치 스코어와 상태가 다시 내려갈 수 있다.
- 매치 스코어, 승자, 세트 집계가 서로 모순된 행이 0건이다.
- 현재 `deriveSetStatus()` 기준과 저장 상태 불일치가 0건이다.
- 세트 화면에서 상위 매치 외 팀을 선택할 수 없다.
- 기본 매치 동기화 버튼이 하나로 정리된다.
- 세트 저장 버튼이 하나로 정리된다.
- 외부 동기화 실패 시 기존 데이터가 부분 삭제되지 않는다.
- 관리자 화면에서 마지막 동기화 결과와 불일치를 확인할 수 있다.
- 타입체크와 추가된 테스트가 통과한다.

## 18. 작업 시 주의사항

- 현재 작업 트리에 존재하는 사용자 변경을 임의로 되돌리지 않는다.
- 관련 없는 공개 페이지나 커뮤니티 UI를 함께 재설계하지 않는다.
- 데이터 복구는 반드시 dry-run부터 실행한다.
- Supabase 쓰기 전 실제 대상 행과 변경값을 출력하고 검토한다.
- 외부 동기화 값이 수동 관리 필드를 `null`로 덮어쓰지 않게 한다.
- 한 화면에서 문제를 숨기는 방식으로 해결하지 말고 저장 데이터와 모든 소비 화면을 함께 맞춘다.
